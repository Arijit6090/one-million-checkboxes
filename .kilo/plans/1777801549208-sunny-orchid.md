# One Million Checkboxes - Technical Architecture & Execution Plan

## Executive Summary

Build a real-time web application allowing users to interact with a grid of 1 million checkboxes with live synchronization across all connected users. The system requires WebSocket-based real-time updates, Redis for state coordination, custom rate limiting, and OIDC/OAuth 2.0 authentication.

## Current State Analysis

### Existing Implementation
- Basic Express + Socket.IO server (`src/index.ts`)
- Redis connection setup (`src/lib/redis.ts`)
- Frontend with lazy-loading checkboxes (`src/public/`)
- In-memory rate limiting (basic, per-socket)
- Redis Pub/Sub for cross-server updates

### Critical Gaps
1. **No authentication** - OIDC/OAuth 2.0 completely missing
2. **Inefficient state storage** - Storing full 1M array as JSON (~10-20MB JSON blob)
3. **Basic rate limiting** - In-memory only, not Redis-backed, resets on restart
4. **No anonymous user handling** - All users treated equally
5. **Missing project structure** - Everything in one file
6. **No .env.example** - Only REDIS_URL mentioned
7. **No README** - Documentation missing

## Phase 2: Efficient Checkbox State Storage (Redis Bitmaps)

### 2.1 Objectives
- Replace inefficient JSON array storage with Redis bitmaps
- Reduce memory from ~10-20MB to ~125KB for 1M checkboxes
- Enable O(1) checkbox read/write operations

### 2.2 Technical Deliverables
- `CheckboxService` class with bitmap operations
- Batch state loading for initial page load
- Atomic checkbox toggle with conflict handling

### 2.3 Implementation Steps
1. Design bitmap storage schema:
   - Key: `checkboxes:state` (bitmap of 1M bits)
   - Key: `checkboxes:stats:checked` (counter)
2. Implement `CheckboxService`:
   ```typescript
   class CheckboxService {
     async getCheckbox(index: number): Promise<boolean>
     async setCheckbox(index: number, value: boolean): Promise<void>
     async getBatchState(start: number, count: number): Promise<boolean[]>
     async getAllState(): Promise<boolean[]>
     async getCheckedCount(): Promise<number>
   }
   ```
3. Use Redis BITGET/BITSET for O(1) operations
4. Use BITCOUNT for efficient checked count
5. Implement batch loading with BITFIELD for ranges

### 2.4 Validation & QA

- [ ] Verify storage efficiency (check Redis memory usage)
- [ ] Test concurrent toggle scenarios

### 2.5 Risk Mitigation
- **Risk**: Race conditions on concurrent toggles
- **Solution**: Use Redis MULTI/EXEC transactions or optimistic locking

## Phase 3: OIDC/OAuth 2.0 Integration with User's OIDC Server

### 3.1 Objectives
- Integrate with user's existing OIDC server using OAuth 2.0 Authorization Code Flow
- Support login flow with proper session management
- Restrict anonymous users to read-only access

### 3.2 Technical Deliverables
- OIDC client integration with user's server
- JWT token validation using JWKS endpoint
- Session management with Redis
- Login/logout routes and frontend UI

### 3.3 Implementation Steps
1. Install dependencies:
   ```bash
   pnpm add jose express-session connect-redis uuid
   pnpm add -D @types/express-session @types/uuid
   ```

2. OIDC Configuration (environment variables):
   - Production: `OIDC_REDIRECT_URI=https://one-million-checkboxes.rithbanerjee.site/auth/callback`
   - Local: `OIDC_REDIRECT_URI=http://localhost:3000/auth/callback`

3. Discover OIDC endpoints dynamically:
   - Fetch `/.well-known/openid-configuration`
   - Get authorization_endpoint, token_endpoint, userinfo_endpoint, jwks_uri

4. Implement OAuth 2.0 Authorization Code Flow:
   ```
   GET  /auth/login      -> Redirect to OIDC server's /o/authenticate
   GET  /auth/callback   -> Exchange code for tokens, store in session
   POST /auth/logout     -> Clear session, optionally revoke token
   GET  /auth/me         -> Return current user info from userinfo endpoint
   ```

5. JWT Token Validation:
   - Fetch JWKS from `/.well-known/jwks.json`
   - Verify tokens using RS256 algorithm
   - Extract user claims (sub, email, name)

6. Session Management in Redis:
   - `sess:{sessionId}` - Session data with access_token, refresh_token
   - Auto-refresh tokens when expired using refresh_token endpoint

7. Implement middleware:
   - `requireAuth` - Blocks unauthenticated requests (for toggle operations)
   - `optionalAuth` - Attaches user if available, allows read access

8. Frontend login flow:
   - Add login button in header
   - Show user info when logged in
   - Anonymous users see "Login to interact" prompt on checkbox click

### 3.4 Validation & QA
- [ ] Login redirects to OIDC server
- [ ] Callback exchanges code for tokens
- [ ] Session persists across refresh
- [ ] Logout clears session
- [ ] Anonymous users see checkboxes but can't toggle
- [ ] Logged-in user info displays correctly

### 3.5 Risk Mitigation
- **Risk**: OIDC server downtime
- **Solution**: Graceful degradation, show error message, allow read-only access
- **Risk**: Token expiration during session
- **Solution**: Use refresh token to obtain new access token automatically

## Phase 4: Custom Redis-Backed Rate Limiting

### 4.1 Objectives
- Implement custom rate limiting without external packages
- Support both HTTP API and WebSocket events
- Use Redis for distributed rate limiting across server instances

### 4.2 Technical Deliverables
- `RateLimiterService` class
- Sliding window rate limiting algorithm
- Per-user and per-IP rate limits

### 4.3 Implementation Steps
1. Design rate limit schema:
   ```
   ratelimit:{type}:{identifier} -> counter with TTL
   ```

2. Implement sliding window algorithm:
   ```typescript
   class RateLimiterService {
     async checkLimit(
       type: 'http' | 'websocket',
       identifier: string,  // IP, userId, or socketId
       maxRequests: number,
       windowMs: number
     ): Promise<{ allowed: boolean; remaining: number; resetAt: number }>
   }
   ```

3. Rate limit tiers:
   - **HTTP API**: 100 requests/minute per IP
   - **WebSocket checkbox toggle**: 
     - Anonymous: 0 (disabled)
     - Authenticated: 10 toggles/second, 500/minute
   - **WebSocket connection**: 5 connections/minute per IP

4. Use Redis Lua script for atomicity:
   ```lua
   -- Atomic increment with TTL set on first increment
   local current = redis.call('INCR', KEYS[1])
   if current == 1 then
     redis.call('PEXPIRE', KEYS[1], ARGV[1])
   end
   return {current, redis.call('PTTL', KEYS[1])}
   ```

5. Integration points:
   - HTTP middleware for API routes
   - Socket.IO middleware for event rate limiting
   - Socket.IO handshake middleware for connection rate limiting

### 4.4 Validation & QA
- [ ] Rate limiting rejects over-limit requests
- [ ] Counter resets after window expires
- [ ] Distributed rate limiting works (test with multiple instances)

### 4.5 Risk Mitigation
- **Risk**: Redis downtime breaks rate limiting
- **Solution**: Fallback to in-memory rate limiting with warning logs

## Phase 5: WebSocket Architecture Enhancement

### 5.1 Objectives
- Secure WebSocket connections with authentication
- Proper connection lifecycle management
- Efficient broadcasting with room-based updates

### 5.2 Technical Deliverables
- Socket.IO middleware for authentication
- Connection tracking in Redis
- Room-based broadcasting for efficiency

### 5.3 Implementation Steps
1. Socket authentication middleware:
   ```typescript
   io.use(async (socket, next) => {
     const token = socket.handshake.auth.token || 
                   socket.handshake.query.token;
     const user = await verifyToken(token);
     if (user) {
       socket.data.user = user;
       socket.join(`user:${user.id}`);
     }
     next();
   });
   ```

2. Track connected sockets in Redis:
   - `connected:{userId}` -> Set of socket IDs
   - `socket:{socketId}` -> User ID and metadata

3. Restructure event handlers:
   ```
   client:checkbox:toggle -> Server validates, updates Redis, broadcasts
   server:checkbox:updated -> Broadcast to all connected clients
   server:rate:limited -> Inform client of rate limit
   ```

4. Implement graceful shutdown:
   - Drain connections on SIGTERM
   - Clean up Redis socket tracking

### 5.4 Validation & QA
- [ ] Unauthenticated users can connect but receive read-only state
- [ ] Authenticated users can toggle checkboxes
- [ ] Connection tracking works for multiple tabs

### 5.5 Risk Mitigation
- **Risk**: Memory leak from unclosed sockets
- **Solution**: Implement heartbeat/ping-pong, cleanup stale connections
